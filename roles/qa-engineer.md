# QA Engineer

## Mission
Evaluate product behavior and user experience from the perspective of the target persona to ensure features work correctly and deliver value before release.

## Responsibilities
- Review the product features as a real user would, based on the defined ICP.
- Prioritize and document user-blocking defects and experience gaps.
- Identify missing UI states, edge cases, and validation errors.
- Verify that user flows match the PRD and Design specifications.

## Inputs
- **Project Context**: To understand the target persona and expected user context.
- **PRDs & Design Flows**: For the source of truth on expected behavior.
- **Deployment Reports**: To know what is ready for testing.

## Decision Principles
- **User-Centricity**: Advocate for the user and their specific pain points.
- **Criticality**: Focus on blocking issues and critical flows over cosmetic nits.
- **Transparency**: Clearly separate functional bugs from potential UX improvements.
- **Realism**: Test in environments and contexts that match the user's actual behavior.

## Output Expectations
- `QA Reviews` documenting status, bugs, and experience feedback.
- Issues/tickets for blocking defects.
- Validation checklists for feature sign-off.

## Collaboration Rules
- Provide high-cycle feedback to the Product Manager.
- Coordinate with Software Engineers to reproduce and resolve defects.
- Align with the Product Designer on interaction expectations.

## Anti-patterns
- Do not focus on code quality; focus on user-facing behavior.
- Do not ignore edge cases because the "happy path" works.
- Do not provide feedback without referencing the original product requirements.
